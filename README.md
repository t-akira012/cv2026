### ②新聞社データ配信プロジェクト(2020)

某大手通信社のスポーツニュース配信システムのAWSインフラ基盤開発。

IBM製オンプレミス業務システムの、Webリプレイスとなる。

#### システム構成

- インフラ: AWS、Docker
- Backend: Python, Django

#### 担当箇所

インフラ開発を方式設計から構築、実装、運用設計まで担当。

- AWS基盤構築
- Dockerコンテナ設計・実装
- CI/CD設計・構築

#### 該当システムの特徴

ほぼ同一のコードを持つシステムが対象スポーツ別に3サービス作成されているという特色を持つ。

加えて、通信社の特殊要件から以下の基盤対応を求められた。

- 災害対策環境として通信社内部のオンプレミス環境での動作させること
- 開発者のローカル環境での動作させること
- AWS環境は本番、ステージングを構築し、ステージングは廃棄用意性を保つこと

#### 対応内容

GitHubをコードベースとし、AWS ECS FargateにデプロイするCI/CDパイプライン・AWS基盤の作成が主な対応となります。

私の参画前にはAWS・Dockerに関する非専任担当者がおらず、
DockerコンテナやAWS基盤は非専任担当者が作成していたため、DevOpsを行うに適さないコードの修正から対応に当たることになりました。

主には、外部化されるべき環境変数のハードコーディングの外部化や、
コンテナ間通信を環境差分（具体的にはAWS ECSでのawsvpc/defaultモード差分）で動作させるための修正を実施しました。

従って、まずはHeroku 12 Factor Appに基づいたコンテナ設計を解説し、コーディング方針をチーム内に周知し、
それと並行して、Backend開発チームの修正中に、AWSクラウド基盤を並行して作成いたしました。

更に、プロジェクトで作成するアプリケーションはスポーツ毎に３サービスあり、
本番／ステージング対応も求められたことから、作成したAWSクラウド基盤の複製作業に入りました。

##### AWS基盤作成

実際に使用したサービスは以下のとおりです。

- RDS
- VPC Endpoint
- SQS
- S3
- CodeBuild
- ECS Fargate
- ECR
- ALB
- ACM
- WAF
- AWS Shield Standard
- EFS

AWS基盤のうち、以下に限りCloudFormation対応を実施しました。

以下のような分類で、インフラのコード化を実施しました。

1. 恒久的な構築

サービス終了まで破棄しないサービス。

- SQS
- ALB
- ECR

1. 廃棄用意性を保った構築

ステージング環境等、頻繁にAWS環境自体を破棄、生成する必要があるサービス。

- ALB TargetGroup
- ECS Fargate

#### 技術的な工夫

1つのサービス用に作られたDockerコンテナを、
AWS基盤、オンプレミス環境、ローカル開発環境
という３環境での動作を求められた現場です。

Heroku 12 Factor Appで提唱される移植性の最大化を実現するために、
・チーム内にベストプラクティスを周知して実際に行う工夫
・docker-entrypoint.sh等で環境差分を吸収する工夫
を実施しました。

更に、AWSインフラ基盤は当初は１サービス向けに作成したものを、
３サービスかつ、本番／ステージングという６環境に複製することになりました。

従って、
・CloudFormationでのコード化を行った後に、環境を6環境に複製
・CloudFormationをAWS CLIで発行するShell Scriptを作成し、生成／廃棄を容易化
・環境を破棄しても、DNSと紐付くALBエンドポイントは変更されないように調整
といった工夫を行いました。

その一方で、スケジュール的にタイトであったことから
全てのサービスをCloudFormation化するのではなく、
一部はコンソールでの構築を行い、作業手順書で再現性を担保する対応も行っています。

### 案件概要

某自治体の災害対策Mapアプリの開発となる。

自治体ながらAWS ECSをコードベースとして利用し、React + TypeScriptで構築されたモダンなアプリケーションとなる。

#### システム構成

- インフラ: AWS, Docker
- Back-end: （自治体における業務サービスにつき詳細非開示、APIのみ開示）
- Front-end: React, TypeScript

#### 担当箇所

インフラエンジニア件Webディレクターとして参画。

##### インフラエンジニアとしての担当箇所

- Dockerコンテナ設計
- AWS ECSコンテナ構成設計
- 負荷テスト（Jmeter）
- nodejs 環境構築（webpack, react, typescript）

##### Webディレクターとしての担当箇所

- 社内ウェブサービス（Slack, GitHub, scrapbox, G Suite）を利用したアジャイル開発フローの構築
- フロントエンジニア向け工程管理
- テストエンジニア向け工程管理
- デザイナー向け工程管理
- 画面設計仕様書の作成
- テスト仕様書の作成
- 契約締結〜納品

#### 対応内容

本来エンジニアとしての参画を希望していたものの、
インフラとマネジメント、画面設計のできる担当者が私に限られていた都合から、Webディレクター件インフラエンジニアとしての動きをした案件となります。

ゼロからの開発となった案件です。

課題として、開発体制が存在しないところから開発に入ったため、以下の順序で対応を行いました。

1. アジャイル開発体制・コミュニケーションフローの構築
2. （デザイナチームとの連携）画面設計書の作成
3. （発注元会社との連携）Docker設計及びECS向けデプロイ対応
4. （テストチームとの連携）テストフローの作成
5. Jmeterを用いた負荷テスト

#### 1. アジャイル開発体制・コミュニケーションフローの構築

まず、アジャイル開発体制の構築からスタートしました。

Slackとscrapboxを利用したコミュニケーション、ドキュメント共有フローを作成。

エンジニアは全員がSlack, GitHubに慣れていることから、チャンネルを
・ドキュメント等を連携する main ch
・エンジニア同士のコミュニケーションを行うdev ch
・デザイナーとの連携を行う design ch
・当日タスクを連携するTodo ch
・GitHub notifyを受信する github ch
で作成し、コミュニケーション体制を構築しています。

全体としてはSlackをメインとしてプルリクレビュー通知等を Notify で通知するアジャイル開発体制となります。

#### 2. （デザイナチームとの連携）画面設計書の作成

そもそも画面仕様等もがないことから、画面設計書を作成しデザイナチームとの連携を実施しました。

こちらはオリジナリティより保守的であることを求められる自治体案件である都合上、他自治体の類似サービスに基づいて
画面設計を作成しています。

#### 3. （発注元会社との連携）Docker設計

Reactアプリケーションを動作させるためのDockerコンテナ設計を実施しました。

他社のAWS環境が本番であることからCI/CDパイプラインを敷けず、
ビルド可能な状態のDockerfileを納品することが求められたため、
npm scriptでビルドを行ったReactAppを、Docker build可能な状態で生成する
Shell Scriptを作成する対応を行いました。

ECS向けの展開では、社内AWS環境に一時的に上記Dockerコンテナをデプロイし、
動作確認後に他社向けにDockerfileを展開する対応を行っています。

#### 4. （テストチームとの連携）テストフローの作成

開発中に issues が最大40件ほど蓄積する状況になったことから、
フロントエンジニアチーム→テストチームでのテスト体制を高速で回すため、
GitHub Projectsを利用したKanban方式を導入、テスト体制を構築しました。

#### 5. jmeterを用いた負荷テスト

ステージング環境にデプロイ後、jmeterを社内PC数台に導入し、シナリオを作成。
負荷テストを行いました。

#### プロジェクトの成果

本来はエンジニアとしての参画を希望してながらもマネジメントポジションでゼロからの開発案件を回すことが求められながらも、
エンジニアとしての開発フローの知見、勘所を発揮することでアジャイル開発を成功させた案件となります。

会社間の制約からCI/CDパイプラインを敷くことはできず、残念ながらDevOpsの技術的な知見を最大限発揮することは叶わなかったのですが、
react+webpack+typescriptのフロントエンドアプリケーションにnpm scriptを導入、Shell Scriptと連携してDockerコンテナ化までを行い、ECSで動作させることを実現しました。
