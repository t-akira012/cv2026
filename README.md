某大手通信社のスポーツニュース配信システムの AWS インフラ基盤開発。

#### システム構成

- インフラ: AWS、Docker
- Backend: Python, Django

#### 担当箇所

インフラ開発を方式設計から構築、実装、運用設計まで担当。

- AWS 基盤構築
- Docker コンテナ設計・実装
- CI/CD 設計・構築

#### 該当システムの特徴

ほぼ同一のコードを持つシステムが対象スポーツ別に 3 サービス作成されているという特色を持つ。

加えて、通信社の特殊要件から以下の基盤対応を求められた。

- 災害対策環境として通信社内部のオンプレミス環境での動作させること
- 開発者のローカル環境での動作させること
- AWS 環境は本番、ステージングを構築し、ステージングは廃棄用意性を保つこと

#### 対応内容

GitHub をコードベースとし、AWS ECS Fargate にデプロイする CI/CD パイプライン・AWS 基盤の作成が主な対応となります。

私の参画前には AWS・Docker に関する非専任担当者がおらず、
Docker コンテナや AWS 基盤は非専任担当者が作成していたため、DevOps を行うに適さないコードの修正から対応に当たることになりました。

主には、外部化されるべき環境変数のハードコーディングの外部化や、
コンテナ間通信を環境差分（具体的には AWS ECS での awsvpc/default モード差分）で動作させるための修正を実施しました。

従って、まずは Heroku 12 Factor App に基づいたコンテナ設計を解説し、コーディング方針をチーム内に周知し、
それと並行して、Backend 開発チームの修正中に、AWS クラウド基盤を並行して作成いたしました。

更に、プロジェクトで作成するアプリケーションはスポーツ毎に３サービスあり、
本番／ステージング対応も求められたことから、作成した AWS クラウド基盤の複製作業に入りました。

##### AWS 基盤作成

実際に使用したサービスは以下のとおりです。

- RDS
- VPC Endpoint
- SQS
- S3
- CodeBuild
- ECS Fargate
- ECR
- ALB
- ACM
- WAF
- AWS Shield Standard
- EFS

AWS 基盤のうち、以下に限り CloudFormation 対応を実施しました。

以下のような分類で、インフラのコード化を実施しました。

1. 恒久的な構築

サービス終了まで破棄しないサービス。

- SQS
- ALB
- ECR

1. 廃棄用意性を保った構築

ステージング環境等、頻繁に AWS 環境自体を破棄、生成する必要があるサービス。

- ALB TargetGroup
- ECS Fargate

#### 技術的な工夫

1 つのサービス用に作られた Docker コンテナを、
AWS 基盤、オンプレミス環境、ローカル開発環境
という３環境での動作を求められた現場です。

Heroku 12 Factor App で提唱される移植性の最大化を実現するために、
・チーム内にベストプラクティスを周知して実際に行う工夫
・docker-entrypoint.sh 等で環境差分を吸収する工夫
を実施しました。

更に、AWS インフラ基盤は当初は１サービス向けに作成したものを、
３サービスかつ、本番／ステージングという６環境に複製することになりました。

従って、
・CloudFormation でのコード化を行った後に、環境を 6 環境に複製
・CloudFormation を AWS CLI で発行する Shell Script を作成し、生成／廃棄を容易化
・環境を破棄しても、DNS と紐付く ALB エンドポイントは変更されないように調整
といった工夫を行いました。

その一方で、スケジュール的にタイトであったことから
全てのサービスを CloudFormation 化するのではなく、
一部はコンソールでの構築を行い、作業手順書で再現性を担保する対応も行っています。

ECS 向けの展開では、社内 AWS 環境に一時的に上記 Docker コンテナをデプロイし、
動作確認後に他社向けに Dockerfile を展開する対応を行っています。
